## Issue 1: `GetMin` has no postcondition

**30 mutants** | Confidence 90–100%

### Spec weakness

`GetMin` carries no `ensures` clause at all. Its contract is empty beyond the implicit well-typedness check. The verifier has nothing to refute when the return value is wrong, arbitrarily negated, or replaced with a constant zero.

There are two distinct sub-weaknesses that together account for all 30 mutants:

#### 1a. Return value in non-empty cases is completely unconstrained

Any mutation that changes what the method returns in the `Node` cases passes verification trivially. Representative surviving mutants:

| Operator | Mutation | Effect |
|----------|----------|--------|
| `AOI` (`__1005-1010`, `__1155-1155`, `__1189-1193`, `__947-952`) | Negates the returned value (`-value`, `-minval`, `-res`) | Returns the negation of the minimum |
| `EVR`/`MRR` (`__1005-1010`, `__1010`, `__1109`, `__1193`) | Replaces return with `0` (default `int`) | Returns zero regardless of tree content |
| `SDL` (`__1093-1120`) | Deletes `minval := GetMin(tree.left)` | `minval` stays at `tree.value`; skips recursion entirely |
| `SWS` (`__1128-1174`) | Swaps the recursive assignment with the `tmp` construction | `tmp` is built before `minval` is updated, so returns `tree.value` |
| `MVR`/`MAP` (`__1109`, `__1115`, `__1143`, `__1149`) | Replaces recursive call result or `Node` constructor with `tree.left` or `tree` | Returns a subtree reference instead of the minimum integer |
| `DCR` (`__866`) | Changes `Node(Empty, …)` pattern to `Empty` | All cases fall through to the recursive `Node(left,…)` branch, which computes incorrectly |
| `SDL` (`__963-1015`) | Deletes the `Node(Empty, value, right)` case | Nodes with only a right child fall into the four-argument `Node` branch and recurse left into `Empty`, returning 0 |

#### 1b. Base-case return value for `Empty` is arbitrary

```dafny
case Empty => res := 0;
```

Returning `0` for an empty tree is a sentinel convention, but it is not specified as such. Mutants that change this to `-1` or `1` (`__898_LVR_-1`, `__898_LVR_1`) both verify, because the spec places no constraint on what `GetMin(Empty)` should equal.

### Suggested fix

```dafny
method GetMin(tree: Tree) returns (res: int)
  requires tree != Empty
  ensures BinarySearchTree(tree) ==> minValue(tree, res - 1)  // res is ≤ every element
  ensures BinarySearchTree(tree) ==> !minValue(tree, res)     // res is actually in the tree
```

Or more directly, linking to the existing predicates:

```dafny
  requires tree != Empty
  ensures BinarySearchTree(tree) ==>
    (exists t :: t in TreeElements(tree) && res == t &&
     forall u :: u in TreeElements(tree) ==> res <= u)
```

#### 2b. Base-case return value for `Empty` is arbitrary

`__1293_LVR_-1` and `__1293_LVR_1` change `GetMax(Empty)` from `0` to `-1`/`1` and both survive.

### Suggested fix

```dafny
method GetMax(tree: Tree) returns (res: int)
  requires tree != Empty
  ensures BinarySearchTree(tree) ==>
    maxValue(tree, res + 1) && !maxValue(tree, res)
```

## Issue 3: `insert` does not guarantee the value is inserted

**25 mutants** | Confidence 80–100%

### Spec weakness

`insert` and `insertRecursion` carry postconditions, but none of them assert that the inserted value actually ends up in the result tree:

```dafny
method insert(tree: Tree, value: int) returns (res: Tree)
  requires BinarySearchTree(tree)
  ensures BinarySearchTree(res)          // ← only structural, not functional

method insertRecursion(tree: Tree, value: int) returns (res: Tree)
  requires BinarySearchTree(tree)
  ensures res != Empty ==> BinarySearchTree(res)
  ensures forall x :: minValue(tree, x) && x < value ==> minValue(res, x)
  ensures forall x :: maxValue(tree, x) && x > value ==> maxValue(res, x)
  // ↑ These only say existing bounds are propagated, not that `value` is present
```

A tree that ignores the insertion entirely still satisfies `BinarySearchTree(res)` (it just returns the original). The `minValue`/`maxValue` postconditions only preserve existing bounds for elements already in the tree — they say nothing about `value` itself being present.

#### 3a. Insertion is bypassed entirely

These mutations cause the method to return the unmodified `tree` for all inputs:

| Mutant | Mechanism |
|--------|-----------|
| `__2235_BBR_true` | Forces `if value == tree.value` to always be `true`; returns `tree` for every non-empty input |
| `__2235_ROR_Ge`, `__2235_ROR_Le` | Changes equality check to `>=` or `<=`; suppresses insertion for half the integer space |
| `__2250-2279_CBE_` | Extracts `return tree` as the only surviving branch; all non-empty trees are returned unchanged |
| `__1771_MAP_0`, `__1771_MVR_tree` | Returns the `tree` argument instead of the `insertRecursion` result |

#### 3b. Insertion into subtrees is bypassed

| Mutant | Mechanism |
|--------|-----------|
| `__2342_MAP_0`, `__2348_MAP_0` | Replaces `insertRecursion(tree.left, value)` with `tree.left` (no insertion into left subtree) |
| `__2481_MAP_0` | Same for the right subtree |
| `__2381_MAP_0`, `__2381_MVR_temp`, `__2381_MVR_tree` | Returns subtree instead of reconstructed node, losing the current node |
| `__2521_MAP_0`, `__2521_MVR_temp`, `__2521_MVR_tree` | Same for the right branch |
| `__2405_MAP_0` | Returns `tree.left` instead of the rebuilt node |
| `__2545_DCR_Empty` | Replaces `temp` (updated right subtree) with `Empty`, dropping all right-side insertions |

#### 3c. Wrong value is inserted

| Mutant | Mechanism |
|--------|-----------|
| `__1778-1778_AOI_` | Inserts `-value` instead of `value` |
| `__1778-1778_EVR_int` | Inserts `0` instead of `value` |
| `__2243_MVR_value` | Changes `if value == tree.value` to `if value == value` (always true); short-circuits all insertions |

#### 3d. Node construction errors

`__2115_DCR_Empty`, `__2151_MVR_tree`, `__2267_DCR_Empty` corrupt the `Empty` base case or the existing-value case, causing missing or empty-node results.

### Suggested fix

```dafny
method insertRecursion(tree: Tree, value: int) returns (res: Tree)
  requires BinarySearchTree(tree)
  ensures BinarySearchTree(res)
  ensures Contains(res, value)                            // ← the missing guarantee
  ensures forall x :: Contains(tree, x) ==> Contains(res, x)  // values preserved
```

where `Contains` is a predicate to be added:

```dafny
predicate Contains(tree: Tree, v: int) {
  match tree
  case Empty => false
  case Node(left, value, right) => value == v || Contains(left, v) || Contains(right, v)
}
```

## Issue 4: `delete` has no postcondition

**53 mutants** | Confidence 90–100%

### Spec weakness

```dafny
method delete(tree: Tree, value: int) returns (res: Tree)
  requires BinarySearchTree(tree)
  // ← no ensures at all
```

The `delete` method requires a valid BST on input but makes no guarantees about its output. Three distinct categories of mutations all survive as a result.

#### 4a. Target value is not deleted

Mutations that change the comparison logic so that the correct subtree is never searched:

| Operator | Mutants | Effect |
|----------|---------|--------|
| `ROR` | `__2873_ROR_{Ge,Gt,Le,Neq,Eq}`, `__3004_ROR_{Eq,Ge,Le,Lt}` | Replaces `<` or `>` in branch conditions; routes to wrong subtree |
| `BBR` | `__2873_BBR_{true,false}`, `__3004_BBR_{true,false}` | Hard-codes branch selection; forces recursion into always-left or always-right |
| `COI` | `__2997-3016_COI_` | Negates `value > tree.value`; misroutes the right-subtree case |
| `EVR`/`AOI`/`MRR`/`MVR` | `__2867`, `__2875`, `__2880` (multiple) | Replaces `value` or `tree.value` in the comparison with `0` or negation |
| `CBE` | `__2886-2987_CBE_`, `__3017-3118_CBE_`, `__3125-3447_CBE_` | Extracts one branch of the if-chain and deletes the others |
| `SDL` | `__3017-3118_SDL_` | Deletes the `value > tree.value` branch entirely |
| `DCR` | `__2772_DCR_Empty` | Makes delete always match `Empty`, returning the unmodified tree |

#### 4b. BST invariant is not preserved in the output

Mutations that corrupt the reconstructed `Node` after deletion survive because there is no postcondition asserting `BinarySearchTree(res)`:

| Mutant | Effect |
|--------|--------|
| `__2910_MVR_tree` | Replaces `delete(tree.left, value)` result with `tree`, creating a cycle |
| `__2950_SAR_2973` | Swaps left and right children in the reconstructed node |
| `__2950_VER_tree` | Replaces `temp` (updated left) with `tree` in the constructor |
| `__2956-2961_AOI_`, `__2956-2961_EVR_int`, `__2961_MRR_int`, `__2961_MVR_value` | Corrupts `tree.value` in the reconstructed node (negates it, replaces with `0` or with the deletion target) |
| `__2973_MVR_temp`, `__2973_MVR_tree` | Places the updated left subtree in the right child slot, or creates a cycle |

#### 4c. Single-child deletion logic is wrong

Mutations in the `tree.left == Empty` / `tree.right == Empty` guard cases:

| Mutant | Effect |
|--------|--------|
| `__3149_BBR_{true,false}`, `__3149_ROR_Neq` | Disables or inverts the no-left-child guard |
| `__3219_BBR_{false,true}`, `__3219_ROR_Neq` | Disables or inverts the no-right-child guard |
| `__3204-3267_SDL_`, `__3241-3257_SDL_` | Deletes the `return tree.left` statement for the left-only case |
| `__3158-3197_CBE_`, `__3229-3267_CBE_` | Extracts only the `return tree.right` or `return tree.left` arm, discarding two-child logic |

#### 4d. Recursive calls produce wrong results

| Mutant | Effect |
|--------|--------|
| `__2910_MAP_0` | Returns `tree.left` (no-op) instead of `delete(tree.left, value)` |
| `__2922-2922_AOI_`, `__2922-2922_EVR_int` | Passes `-value` or `0` to the recursive call instead of `value` |
| `__2949_MAP_0`, `__2949_MVR_temp`, `__2949_MVR_tree` | Returns the left subtree alone instead of rebuilding `Node(tree.left, …, tree.right)` |

### Suggested fix

```dafny
method delete(tree: Tree, value: int) returns (res: Tree)
  requires BinarySearchTree(tree)
  ensures BinarySearchTree(res)                                 // ← preserve structure
  ensures !Contains(res, value)                                 // ← target is removed
  ensures forall x :: Contains(tree, x) && x != value ==> Contains(res, x)  // ← rest preserved
```

## Issue 5: Traversal methods are entirely unspecified

**5 mutants** | Confidence 100%

### Spec weakness

Both `Inorder` and `Postorder` have no `requires` or `ensures` clauses:

```dafny
method Inorder(tree: Tree)   { … }   // no spec
method Postorder(tree: Tree) { … }   // no spec
```

Because neither method has any contract, any alteration to their behaviour — including silently omitting recursive calls — passes verification trivially.

#### Surviving mutations

| Mutant | Effect |
|--------|--------|
| `__3561-3579_SDL_` | Deletes `Inorder(tree.left)` — left subtree is never traversed |
| `__3617-3636_SDL_` | Deletes `Inorder(tree.right)` — right subtree is never traversed |
| `__3752-3772_SDL_` | Deletes `Postorder(tree.left)` — left subtree silent |
| `__3780-3801_SDL_` | Deletes `Postorder(tree.right)` — right subtree silent |
| `__3780-3801_SWS_` | Swaps `Postorder(tree.left)` and `Postorder(tree.right)` — traversal order inverted |

### Note

Specifying traversal order in Dafny requires ghost output (e.g., returning a sequence). These methods currently use `print` side effects, which are opaque to the verifier. The fix requires refactoring into a functional form:

```dafny
method Inorder(tree: Tree) returns (result: seq<int>)
  requires BinarySearchTree(tree)
  ensures forall i :: 0 <= i < |result| - 1 ==> result[i] < result[i+1]  // sorted
  ensures forall x :: Contains(tree, x) ==> x in result                  // complete
```

## Cross-cutting observation: `GetMin` is called inside `delete`

The two-child deletion case uses `GetMin(tree.right)` to find the in-order successor:

```dafny
var minVal := GetMin(tree.right);
temp := delete(tree.right, minVal);
res := Node(tree.left, minVal, temp);
```

Because `GetMin` has no postcondition, the verifier cannot prove that `minVal` is actually the minimum of `tree.right`, or even that it is present in `tree.right`. This means:

1. Issues 1 and 4 compound: even if `delete` were given postconditions, the verifier could not discharge them without also fixing `GetMin`.
2. The cluster of `GetMin`-related WEAK\_SPEC mutants inside `delete` (e.g., `__3204-3267_SDL_`, the node-reconstruction mutations) partially overlaps with Issue 4 — the root cause is shared.

A complete fix therefore requires addressing Issues 1 and 4 together before the verifier can fully check `delete`.
